Duplicate Query in Dynamic Sampling Rule Processor

A duplicate query in the dynamic sampling rule processor was issuing hundreds of millions of daily queries. This was done despite the fact that the data already resided in memory and could be accessed without a query. Pull request #115792 was opened to remove this unnecessary query.

Discovery

The performance issue was discovered using Sentry's tracing tool. Using the "aggregates" table and sorting by high span-emitting traces, I was able to triage subroutines by volume. Clicking into individual traces showed that a database query to the "sentry_project" table was being emitted despite a cache lookup to the "project:<id>" cache key being emitted. Inspecting the code location revealed the defect.

Results

After merging the pull request, the dynamic sampling rules processing subroutine stopped emitting database queries. The program itself reduced its queries to the "sentry_project" table to near zero. Total database time saved was a little over eight hours. Total application time saved was a little under a week.
