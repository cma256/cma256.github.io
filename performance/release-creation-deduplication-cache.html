Release Creation Deduplication Cache

Published on Tuesday, May 26, 2026

On every event received, the system attempted to create a release and "boost" its dynamic sampling profile. This required between 5 and 11 Redis requests per message. The operation was only necessary when processing a release for the first time. The first message would successfully write, and every request that followed would hit Redis and discard the results. Because the system issued many Redis requests and because these requests formed a data dependency chain, this produced a persistent and unnecessary request waterfall. I opened pull request #116173 to remove these unnecessary attempts. It uses a simple LRU cache and retains seen releases in memory. A "max_size" parameter bounds the size of the cache. Hashing the cache key to an integer of known size bounds memory to "max_size" multiplied by 100 bytes.

Discovery

Sentry's tracing tool surfaced the performance issue. Using the "aggregates" table and sorting by high span-emitting traces, I triaged subroutines by volume. Clicking into individual traces showed the span waterfall. A priori knowledge of the system gave me the intuition that this was not necessary.

Results

After merging the pull request, processing time declined by 50% with a warm cache. A persistent 20% decline in CPU usage was observed. Memory usage was unchanged despite retaining up to 120MB of cache per machine. Fifteen billion average daily cache requests were avoided with this change representing 175,000 requests per second.
